# 決済 reconcile(不整合検知)の実装パターン

webhook の取りこぼし・逆順配送・諦めた再配送で残る「決済プロバイダと自前 DB のずれ」を、日次で突き合わせて直す。Stripe を例にする。

## 設計

- 正本はプロバイダ API の現在状態(customer ごとの subscriptions.list 等)。イベントの再生ではなく状態の突き合わせにする
- 現在状態 → あるべき entitlement の変換は、webhook と同じ status 対応の純関数にする。webhook 処理と別の解釈を作ると二重定義になる
- 修復は webhook と同じ書き込み経路(順序ガード付きの単文 RPC 等)を再利用する。新しい書き込み経路を作ると競合解決の二重実装になる
- occurredAt には取得開始時刻を使う。取得〜書き込みの間に届いた webhook はより新しい時刻を持つため順序ガードで勝ち、reconcile が新しい状態を巻き戻さない
- 触ってよい行を限定する。対象プロバイダ由来(source)の行のみ、期限は伸長のみなど、webhook 側の不変条件を踏襲する
- 手動剥奪(revoked 等)の行を reconcile が自動復活させないかを確認する。webhook との対称性を保って上書きを許すか、剥奪をスキップして対称性を崩すかは運用判断で、どちらを選んだかを実装に明記する
- 1 customer の失敗で全体を止めない。失敗が残れば 500 を返して cron の run を赤くする。修復時は何をどちらへ直したかをログに残す
- 救えないケースを明示する。customer リンク自体が失われたユーザーは reconcile の対象外で、entitlement の期限による自然失効(fail-closed)が最後の網になる
- customer の走査はページングで全件辿る(1 ページ上限に暗黙依存しない)

## トリガ

- デプロイ先が未定なら GitHub Actions schedule + workflow_dispatch から公開エンドポイントを叩く形がデプロイ非依存。デプロイ先確定後にプラットフォームの cron へ移す判断ができる
- エンドポイントは cron との共有 secret を Bearer で受け、定数時間比較(timingSafeEqual)で検証する
- workflow は secrets 未設定なら明示エラーで赤くする。黙って成功させると監視にならない

## 検証

- 変換とずれ判定の純関数を unit で網羅する(支払済み・猶予・終了・DB 行なしの各方向)
- route は認可(401)と部分失敗(500)を含める
- fake プロバイダに「fake 世界の定義」を決めて状態照会を実装する(例: 解約は存在しない世界とし、entitlement 行を active な subscription として映す)。定常状態では reconcile が no-op になり並列 E2E を乱さない。E2E は DB 側だけ壊した行が修復される方向を検証し、プロバイダ側が進んでいる方向は unit で担保する
