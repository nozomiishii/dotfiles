# API レート制限の実装パターン

外部サービス(Upstash 等)を増やさず、DB(Postgres)を正本にするユーザー単位の固定窓カウンタ。サーバーレス等のマルチインスタンス環境でもカウントを共有できる。

## 対象の選定

- 明示的にコストが漏れる面(外部 API のトークン発行・生成系・重い計算)を先に挙げ、さらに「裏でそれらを起動するルート」も洗い出して含める(作成 API が非同期で生成処理を起動する構成が典型)
- 全対象が認証必須ならキーはユーザー ID で足りる。IP キーは共有 NAT の巻き添えがあるため認証前の面にだけ検討する
- トークン発行系(音声 SDK の一時トークン等)は発行レートを絞れるだけで、発行済みトークンでの利用量そのものは絞れない。外部サービス側のクォータ・使用量アラートを別レイヤーとして設定する

## 設計

- カウントは upsert 単文の DB 関数(RPC)で行う。読んでから書く方式は並行リクエストで取りこぼす
- 固定窓: 窓が過ぎていれば起点と件数をリセット、過ぎていなければ加算。超過後も件数は進めるが窓の起点は変えない(連打で窓が伸び続けない)。戻り値は「許可なら 0、超過なら再試行までの秒数」
- 窓境界で理論上最大 2 倍のバーストが通るのは固定窓のトレードオフ。sliding window の複雑化に見合うかで判断する
- 期限切れ行の定期削除は持たない。次の消費で数え直すため、行数はアクション数 × ユーザー数で頭打ち
- テーブルと関数はサーバー専用にする: クライアントロールから revoke、RLS 有効化(policy なし)、関数は既定で public に付く execute を剥がす
- 判定不能(RPC 失敗)時の fail-closed / fail-open はアクションの性質で決める。コスト保護面は closed(429)に倒す
- 応答は 429 + Retry-After(秒)。上限値の定数は 1 ファイルに集約し、初期値は仮置きであることを明示する(実利用データで調整)

## 検証

- unit: RPC の呼び出し引数、許可 / 超過 / RPC 失敗時の分岐、429 応答の形。ルート側は 429 時に本処理(外部 API・DB 書き込み)へ到達しないことまで見る
- SQL 関数は psql で直接検証する: 上限まで許可 → 超過で残秒数 → 窓経過後に再許可。クライアントロール(set role)からの実行が permission denied になること
- E2E: 上限回数まで 200、次で 429 + Retry-After
