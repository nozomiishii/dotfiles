# /money

支払いの前が一番安くできる。契約・購入の手続き案内やプラン比較を出す前に、このチェックを通す。作業の途中で課金が選択肢に入ったときも同じ。

## 聞くより並べる

割引が使えるかをユーザーへの質問で確かめない。学生か、事業者か、と聞かれ続けるより、適用時の価格と条件が並んでいるほうが早く選べる。質問は、初回契約か既存かのように、回答で提示が大きく変わる 1 点に絞る。

## 安い道の探索

公式の料金ページに出ている価格とキャンペーンだけで比較を終えない。以下を WebSearch で確認し、定価と並べる。

- 無料枠・非商用ライセンス。そもそも払わずに足りるか
- 学割・教職員割の適用時価格。開発者向けサービスなら GitHub Education の Student Developer Pack (<https://education.github.com/pack>) の対象かも見る。Adobe の認定スクールのように、講座受講で誰でも学割価格になるルートも含む
- 小規模・個人開発者向けプログラム: Apple の App Store Small Business Program のように、申請しないと適用されない減額
- 正規リセラー・別販路: Amazon のオンラインコード版や公式パートナー経由の価格差
- セールとの距離: 開催中の公式キャンペーンがないかをまず確認する。急ぎの依頼ほど見落としやすく、定価を基準に案内してしまう。恒例セールが近ければ待つ選択肢も出す。プライムデーは 7 月、ブラックフライデーは 11 月、ほかに年末年始・新生活シーズン

## 事業形態で変わる道

事業にも使える支出なら、個人のまま・開業届あり・一人法人で損得がどう変わるかを補足に添える。補助金の対象になるか、事業者・法人プランのほうが安いか、経費にできるか。形態を変えたほうが得になるなら、差額の概算まで書く。

## 提示の形

定価と見つかった安い道を並べ、差額の概算を示す。手続きを止めるのではなく、知った上で選べる状態にしてから進む。
