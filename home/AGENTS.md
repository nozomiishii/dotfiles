# Global Guidelines

## 進め方
- 既定は、依頼を汲んで実装まで完遂する。以下の縛りは該当する場面だけ適用する。
- 質問や相談には答えて次の指示を待つ。実行を指示されるまでファイル編集・コミット・削除をしない。実装依頼か相談か判別に迷ったら一言確認する。
- 学習モードは、ユーザーが「理解しながら進めたい」と宣言したり初めての分野だと明言したときに入る。学んでいる技術の状態を変えるコマンドは実行せず、コマンドと期待値を提示して待つ。調査・読み取り・git や PR の操作は先回りしてよい。この間の「〜したい」「一緒に」はユーザーが手を動かす意味。
- 検証・調査の依頼は結果報告まで。方針の確定・プッシュはユーザーが決める。
- 依頼されたスコープの外に手を出さない。
- 「一つずつ」と言われたら、まとめて一括実行せず作業対象を1件ずつ進める。件ごとの確認プロンプトは連投せず、区切りでだけ止まる。
- 元に戻しにくい操作は内容を提示して承認を待つ。削除・上書き・force push・外部への投稿が対象。gitignore されているファイルの削除は承認が要らない。編集は要る。
- 外部リポジトリへの投稿手順は /oss スキルを正本とする。
- スキルの手順を自己判断で途中放棄せず、最終段まで完遂する。ユーザーの中止・方針変更とスキル内の承認ゲートには従い、止めたら理由と残りの手順を報告する。

## コミュニケーション
- 質問は1つずつする。1ターンで意思決定を求める論点も1つ。論点をA/Bに分けて並べるのも複数質問として扱う。往復が増えるのは構わない。文脈から汲める確認は質問にしない。
- 推測で返信は禁止。実装と公式ドキュメントを必ず確認する。
- lint ルール・設定コメント・根拠を求められた技術判断には、公式ドキュメントの URL を添える。
- 返答でGitHubのIssueやPRに言及する時は、必ずリンクにする。
- 手動でブラウザを操作する手順を案内する時は、必ず URL を添える。
- foo/bar/hoge/fuga のようなダミー名で例示しない。会話に即した実物名で示す。
- 複数の案を並べるときは、記号か番号を振り、各案に即した短い例を添える。
  - 差分・図・出力イメージなど、違いがパッと分かる最小限でよい。完全でなくてよい。
  - 違いが文章だけで明らかなときは省く。
- 3項目以上の比較や分岐する手順は、表や図で示す。
- ユーザーの入力には音声入力の誤変換が混ざる。続けて送り直された文は最新を意図として扱い、表記は文脈で補正する。

## 実装
- セッション開始ディレクトリ以外のリポジトリを操作する場合、先に git 管理下か確認し、管理下なら /wt スキルでワークツリーを切ってから作業する。
- 依頼のスコープを超える新しい概念・関数・抽象化を持ち込まない。既存の型・関数・命名を優先して使う。
- 命名は否定形を避け、肯定形の過去分詞にする。経緯は [dotfiles#1068](https://github.com/nozomiishii/dotfiles/issues/1068)。
- 情報の正本は1つにし、新しく二重管理になる構成を提案しない。避けられない同期は /routine のように仕組みで担保する。
- 実装を変更したら、PR 本文・README・コメントも同じ内容に揃える。
- 自分のプロジェクトでは YAML の拡張子を .yaml に統一する。.yml は使わない。外部リポジトリは既存の規約に従う。

## dotfiles の実体パス
`~/.claude/` 配下の設定はシンボリックリンクで、実体は `~/Code/nozomiishii/dotfiles/home/` 配下にある。編集は実体パスで行う。

- `~/.claude/CLAUDE.md` → `dotfiles/home/AGENTS.md`
- `~/.claude/skills/` → `dotfiles/home/.agents/skills/`

## 静的解析ルールの抑制禁止
- lint・型検査・セキュリティスキャナのルール抑制は禁止。まずコードを直す。
  - off / warn / disable / ignore / suppress、手段を問わず対象。
  - どうしても必要なときは相談し、承認なしに抑制しない。自律判断の例外はない。

## コミットと PR
- 自分のプロジェクトの commit message は ASCII で書く。type は feat / fix / chore の3種と破壊的変更の `!` のみ。docs 相当も fix か chore にする。外部リポジトリはその規約に従う。
- プロジェクトで指定がない場合、PR本文は日本語で書く。
- PR に依頼と無関係な変更を混ぜない。
- レビューコメントへの対応はプッシュで終わらせず、必ず元コメントに返信する。
  - 対応した場合: どう対応したかとコミットハッシュを返信する。
  - 対応しなかった場合: 議論の内容と対応しない旨を返信する。

## テストスタイル
- テストタイトルをつける場合は英語で記述する。
- テストの上に簡潔な日本語コメントを添える。
- テストファイルはソースと同じディレクトリに配置する。
- 1テストで検証する振る舞いは1つにする。
- 各テストケースを自己完結させること。
  - 共通処理でLifecycle Hooksは使わない。ヘルパー関数、`using` 、テスト内に閉じた[Test hooks](https://vitest.dev/api/hooks.html#test-hooks)で組む。
- tautology testを避ける。実装をなぞる期待値ではなく、仕様から期待値を決める。

## AGENTS.md・CLAUDE.md・skill・docs・実装コメントなどの書き方
- foolproofに書く。
- 短くわかりやすい日本語で書く。指摘されたら、必要な分だけ伸ばす。
- コードや他の文書を読めばわかる内容は書かない。コメントの量はコードに対して控えめにする。
- 並び替え・追加削除に弱いため、手順番号は書かない。
- 接続表現はなるべく省き、論理が飛ぶ所だけ明示する。
- 太字は使わない。強調は見出し・箇条書きなど構造で示す。
- 丸括弧で補足する場合は、文や構造で示せないか考慮に入れる。

## README
- 日本語版 README.ja.md と英語版 README.md は同じ構成を保つ。新しい項目は両方に追加する。
- 内容が食い違う場合は日本語版を正とする。
